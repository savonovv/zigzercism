QUESTION 01

How does a mutable byte buffer work?

A []u8 is a mutable view over a sequence of bytes. The slice stores a pointer and a length: it knows where the storage begins and how many bytes are accessible, but not which bytes contain meaningful data.

var storage: [capacity]u8 = undefined;
const buffer: []u8 = &storage;

When building data incrementally, track the initialized portion with a separate cursor:

var used: usize = 0;

// Write new data starting at buffer[used].
// Advance used by the number of bytes written.

return buffer[0..used];

Core rule: buffer.len describes accessible storage. used describes initialized data.

QUESTION 02

How do I write bytes and strings into a buffer?

You can assign one byte by index:

buffer[i] = byte;

A byte slice cannot be replaced with a string using ordinary assignment. Copy the source bytes into a destination slice instead. @memcpy is appropriate when the source and destination do not overlap:

const text = "hello";
@memcpy(buffer[used .. used + text.len], text);
used += text.len;

Not valid: buffer[used .. used + text.len] = text;

Using text.len instead of a hard-coded length keeps the destination size tied to the actual source.

QUESTION 03

How do I expose only the initialized data?

Returning buffer returns its full length, including bytes you did not write. Slicing creates a view containing only the initialized prefix:

const initialized = buffer[0..used];
return initialized;

The returned slice still refers to the original storage. Slicing does not copy bytes or allocate memory, so that storage must remain valid while the returned slice is used.

QUESTION 04

Why does a buffer contain unexpected bytes?

Storage initialized with undefined has no usable initial value. Debug builds may fill untouched memory with a recognizable pattern such as 0xaa, but code must not rely on any particular value.

var storage: [capacity]u8 = undefined;

// Only storage[0..used] becomes valid after writing.

Unexpected trailing bytes usually mean that code exposed more of the buffer than it initialized. Read and return only the region known to contain valid data.

QUESTION 05

Why are buffer-formatting operations fallible?

A formatting function can run out of space, so it returns an error union such as error{NoSpaceLeft}![]u8. A function returning plain []const u8 must handle that error first.

const result = try std.fmt.bufPrint(buffer, "{d}", .{value});
return result;

Use try when the surrounding function can return the error. Use catch when it must recover, translate the error, or when a proven size invariant makes failure impossible.

Version note: std.fmt.bufPrint is the buffer-formatting API in Zig 0.16. Standard-library names can differ between Zig versions.

QUESTION 06

How do I append multiple byte slices?

Keep a cursor and copy each selected piece at that cursor:

var used: usize = 0;

for (parts) |part| {
    @memcpy(buffer[used .. used + part.len], part);
    used += part.len;
}

return buffer[0..used];

A fixed-buffer writer is useful when many operations need to append or format data. It maintains the position and provides operations such as writeAll and print. An explicit cursor is often simpler for a small, fixed set of writes.

QUESTION 07

How do I reverse byte-string data?

To reverse bytes already written into the buffer, reverse only the initialized slice:

std.mem.reverse(u8, buffer[0..used]);
return buffer[0..used];

To copy from another slice in reverse order:

for (source, 0..) |_, i| {
    buffer[i] = source[source.len - 1 - i];
}

return buffer[0..source.len];

Text warning: This reverses bytes and is correct for ASCII. UTF-8 characters may contain multiple bytes and require Unicode-aware handling.

QUESTION 08

Which writing method should I choose?

Method Use it when
buffer[i] Writing or transforming one byte.
@memcpy Copying a complete non-overlapping byte slice.
copyForwards Preferring an explicit standard-library copy operation.
for loop Reversing, filtering, or changing bytes while copying.
Writer Performing many sequential or formatted writes.

Zig strings are byte slices. Copying a string copies its bytes and does not automatically add a null terminator.

QUESTION 09

Why does a function need an allocator?

A function needs an allocator when it must create runtime-sized data, whether that data is temporary or returned to the caller. The allocator is an interface for requesting and releasing memory; it is not the memory itself.

fn transform(
    allocator: std.mem.Allocator,
    input: []const u8,
) std.mem.Allocator.Error![]u8 {
    const output = try allocator.alloc(u8, input.len);

    for (input, 0..) |byte, i| {
        output[i] = std.ascii.toUpper(byte);
    }

    return output;
}

alloc(u8, input.len) requests a mutable byte slice. Allocation can fail, so try propagates errors such as OutOfMemory.

Temporary collections can also use an allocator and free their storage before the function returns. Dynamic containers request more memory as they grow, so the caller does not need to know their final size. If an implementation needs no dynamic storage, an allocator parameter may remain unused.

Ownership

Unless an API documents otherwise, the caller owns a returned allocation and must release it with a compatible allocator:

const result = try transform(allocator, input);
defer allocator.free(result);

// Use result here.

Lifetime mistake: Do not return a slice pointing to a local array. Local storage stops being valid when the function returns. Allocated storage remains valid until its owner frees it.

Ownership rule: The code that receives ownership is responsible for arranging exactly one matching free, usually immediately with defer.

QUESTION 10

How do I associate numeric data with enum values?

An enum represents a closed set of identities. Its backing values must be integers, so an enum cannot directly store a floating-point coefficient. Keep the identity in the enum and map each tag to its data with a method and a switch expression:

const Tier = enum {
    basic,
    premium,

    fn coefficient(self: Tier) f64 {
        return switch (self) {
            .basic => 1.0,
            .premium => 1.5,
        };
    }
};

A switch returns one value for one input. Its cases must be known at compile time, so runtime values from a slice belong in a loop rather than dynamically generated switch cases. Multiple fixed cases can share one result, such as 'a', 'e' => 1.

Exhaustiveness: Zig requires every possible enum tag to be handled. Adding a new tag therefore points directly to mappings that need updating.

QUESTION 11

Why does integer-to-float calculation order matter?

Zig does not implicitly mix integer and floating-point types. Convert an integer explicitly with @floatFromInt, giving the result a concrete float type:

const count: usize = 95;
const count_float: f64 = @floatFromInt(count);
const average = count_float / 3.0;

Convert before division when the fractional part matters. In @floatFromInt(count / 3), integer division happens first, so the fraction is already lost before conversion.

Precision versus formatting

An f64 does not store a fixed number of decimal digits. Output formatting controls only how the value is displayed:

std.debug.print("{d:.2}\n", .{average});

This prints two digits after the decimal point but does not change average. When testing a calculated float, compare it with a tolerance using std.testing.expectApproxEqAbs instead of formatting it or expecting exact equality.

Common mistake: Converting after integer arithmetic changes the result's type, but it cannot restore a fractional part that integer division already discarded.

QUESTION 12

How can loop order prevent duplicate counting?

When several rules can select the same item, iterating by rule can generate that item more than once. Iterating by item instead gives each item one opportunity to contribute:

outer: for (items) |item| {
    for (rules) |rule| {
        if (matches(item, rule)) {
            total += item;
            continue :outer;
        }
    }
}

The labeled continue stops checking rules after the first match and advances the outer loop. This uses constant extra memory because no set of previously seen items is needed.

The tradeoff is time: checking every item against several rules takes work proportional to the number of items times the number of rules. Generating items rule by rule can do less work, but overlapping results then require deduplication, such as a set or a problem-specific mathematical method.

Range boundary: Zig ranges are ascending and end-exclusive. A runtime range such as 1..0 is not an empty descending range and can overflow while iterating. Represent the empty case without placing a larger start before a smaller unsigned end.

QUESTION 13

How do I test that a function returns an expected error?

Pass the function's unresolved error union directly to std.testing.expectError:

try std.testing.expectError(
    error.Invalid,
    parseValue(input),
);

Do not put try around parseValue(input). That would propagate the expected error before expectError could inspect it. The outer try instead handles the assertion's own result: the test fails if the call succeeds or returns a different error.

Mental model: Give expectError the unopened error union. Let the helper verify which branch and which error it contains.

QUESTION 14

When can I omit a struct's type name?

When the expected type is already known from context, begin a struct literal with .{ instead of repeating its type:

const Point = struct {
    x: f64,
    y: f64,
};

fn origin() Point {
    return .{ .x = 0, .y = 0 };
}

The function's return type tells Zig that the literal is a Point. The same inference works in other contexts with a known expected type.

Named fields remain required: Ordinary structs do not support positional initialization. Write .x = value, not just value.

QUESTION 15

How do I tokenize input and allocate an exact-size result?

std.mem.tokenizeAny treats any byte in its delimiter slice as a separator and skips empty tokens created by repeated delimiters:

var iterator = std.mem.tokenizeAny(u8, input, " \t-_");

while (iterator.next()) |token| {
    // token is a non-empty []const u8.
}

next() returns ?[]const u8, not bool. Optional capture unwraps each token and ends the loop when the iterator returns null.

State and allocation

Token iterators are stateful and one-way. After one loop reaches the end, a second loop over the same iterator receives nothing. Recreate the tokenizer when using two passes:

var counter = std.mem.tokenizeAny(u8, input, delimiters);
var count: usize = 0;
while (counter.next()) |_| count += 1;

const output = try allocator.alloc(u8, count);

var tokens = std.mem.tokenizeAny(u8, input, delimiters);
// Fill output from tokens.

A two-pass design avoids guessing the output size: count first, allocate exactly once, then fill. Use usize for allocation lengths and slice indexes because it is the native size type used by Zig's memory APIs.

Uninitialized output: Allocated bytes initially contain no meaningful values. If a consumed iterator makes the write pass skip every assignment, returning the full allocation exposes those uninitialized bytes.

QUESTION 16

How do I print debugging information while running tests?

std.debug.print writes formatted output to standard error without requiring an allocator or error handling. Its output appears alongside the normal zig test results:

std.debug.print(
    "input={s}, index={d}, byte={c}\n",
    .{ input, index, byte },
);

Use {s} for byte strings, {d} for decimal numbers, {c} for characters, and {any} for a general representation. Include labels and a newline so repeated calls remain readable.

When inspecting a partially initialized buffer, print only its valid region:

std.debug.print("result={s}\n", .{buffer[0..used]});

Temporary versus permanent: Use debug prints to investigate control flow and intermediate values, then remove them. Use std.testing assertions for permanent checks because they produce structured failures.

QUESTION 17

How can I compare distances without calculating a square root?

When only the ordering of Euclidean distances matters, compare squared values. Squaring preserves the order of non-negative distances, so the square root is unnecessary:

const distance_squared = x * x + y * y;
const radius_squared = radius * radius;

if (distance_squared <= radius_squared) {
    // The point is inside or on the circle.
}

Direct multiplication is also clearer than a generic power function when the exponent is exactly two. This avoids both sqrt and pow while preserving boundary comparisons.

Flatten classifications with early returns

When checking ordered thresholds, test them from the most restrictive outside case toward the inside and return immediately:

if (value > largest_limit) return outside;
if (value > middle_limit) return outer_band;
if (value > smallest_limit) return middle_band;
return inner_band;

Why it helps: Early returns remove unnecessary else nesting, and squared comparisons avoid computing a value that the classification does not need.

QUESTION 18

How do I process decimal digits and integer powers?

When the input is already an integer, no tokenizer is needed. Remainder by ten reads the last decimal digit, and integer division by ten removes it:

var value = original;
while (value > 0) : (value /= 10) {
    const digit = value % 10;
    // Use digit here.
}

Use a mutable copy so the original remains available after the loop. Handle zero deliberately because the loop body does not run when the initial value is zero.

Powers and shifts

Zig has no numeric exponentiation operator. In Zig 0.16, std.math.pow computes a numeric power. The ** operator repeats arrays and therefore expects a compile-time-known repetition count.

Bit shifting is different: for an integer result that fits, x << n multiplies by 2^n. Thus x << 1 doubles and x << 2 multiplies by four; neither operation squares x. Floating-point values cannot be shifted.

To process an integer one binary digit at a time, mask the lowest bit with value & 1, then remove it with value >>= 1. This is the base-two analogue of reading a decimal digit with remainder by ten and removing it with division by ten.

For a runtime shift, give the left operand a fixed-width integer type and validate the shift amount before subtracting or shifting. A geometric sequence that fills every bit of an unsigned type sums to that type's maximum value: std.math.maxInt(u64) is 2^64 - 1 without attempting to represent the out-of-range value 2^64.

Type rule: Keep accumulators wide enough for the input domain. A usize accumulator is not sufficient for calculations over arbitrary u128 values.

QUESTION 19

When should I replace a loop with a closed-form formula?

A loop is often the clearest first implementation. When a well-known formula exists, it can reduce work from linear time to constant time:

const sum = n * (n + 1) / 2;
const square_of_sum = sum * sum;
const sum_of_squares = n * (n + 1) * (2 * n + 1) / 6;

Name intermediate values so the code preserves the mathematical distinction between a sum and the square of that sum. Returning sum where sum * sum is required can also make later unsigned subtraction underflow.

Overflow tradeoff: Constant-time formulas may create large intermediate products. Divide factors before multiplication where algebra permits, or use a wider checked type when the input range requires it.

QUESTION 20

How do I initialize, iterate, and return an AutoHashMap?

An AutoHashMap manages its own dynamic storage. Initialize one map with an allocator; allocator.alloc(Map, count) would instead create a slice of map objects:

var result = std.AutoHashMap(u8, u32).init(allocator);
errdefer result.deinit();

Insertion can allocate, so put is fallible. The errdefer cleans up partial construction if an insertion fails, but does not run when ownership is successfully returned to the caller.

Iteration

An iterator must be mutable because next() advances its cursor. Entries expose pointers to their stored key and value:

var iterator = map.iterator();
while (iterator.next()) |entry| {
    const key = entry.key_ptr.*;
    const value = entry.value_ptr.*;
}

Iteration order is unspecified. Do not insert or remove entries through the map while its iterator is active.

Ownership and capacity

Use ordinary defer map.deinit() in the scope that owns and consumes the map, so cleanup also runs when a later operation fails. Use errdefer while constructing a value that will be transferred on success.

A map grows automatically, so counting input first is not required. If the final entry count is already known cheaply and the map is large, ensureTotalCapacity can reserve space before insertion. A separate counting pass is a tradeoff: it may reduce reallocations but adds another full traversal.

Optimization rule: For small maps, prefer one clear insertion pass. Reserve capacity when scale or measurement justifies the extra logic.

QUESTION 21

Do I need to store a slice's size separately?

No. A slice stores a pointer and a length, available as slice.len. A separate size field duplicates state and can become inconsistent.

Check for an empty slice before indexing its last element or using an aggregate that requires data:

if (values.len == 0) return null;

const latest = values[values.len - 1];
const greatest = std.mem.max(i32, values);

In Zig 0.16, std.mem.max scans a non-empty slice in linear time and performs no allocation. For pairwise processing, a multi-sequence loop traverses equal-length slices in lockstep:

for (left, right) |a, b| {
    if (a != b) differences += 1;
}

Validate equal lengths before the loop. This gives O(n) time and O(1) extra space, which is optimal when every pair must be examined.

Lifetime rule: Do not return a slice into a local array created inside the function. Returned slices must refer to storage that remains alive, such as caller-owned input, struct-owned storage, or an allocation transferred to the caller.

QUESTION 22

How do I keep a sorted top-k result in a fixed array?

A fixed array has a compile-time capacity, not a runtime empty state. Pair it with a logical length:

var values: [capacity]i32 = undefined;
var used: usize = 0;

Only values[0..used] is initialized. Search that prefix for the insertion position; use values.len only as the physical capacity. After deciding that an item belongs, compute the new length, shift the affected tail right, insert once, and update used.

Backward shift

To insert 100 at index zero in [90, 70, 30], copy from the back so no unread value is overwritten:

90  70  30
90  70  70  // copy index 1 to index 2
90  90  70  // copy index 0 to index 1
100 90  70  // write the new value at index 0
var j = new_len - 1;
while (j > insert_at) : (j -= 1) {
    values[j] = values[j - 1];
}
values[insert_at] = item;

Changing j only moves the cursor; the assignment in the loop body moves the data. A forward shift would overwrite the source for the next copy.

Compile-time capacity

To reuse the same bounded container with different fixed capacities, return a type from a function taking a comptime parameter:

fn Bounded(comptime capacity: usize) type {
    return struct {
        values: [capacity]i32 = undefined,
        used: usize = 0,
    };
}

var small = Bounded(3){};
var larger = Bounded(5){};

Each capacity creates a distinct type. Inside it, values.len is the capacity, so the algorithm does not repeat a hard-coded number. A runtime capacity instead requires allocated slice storage and an ownership policy.

Invariant: values[0..used] is initialized and sorted. values[used..values.len] is never read. The logical length never exceeds capacity.

For capacity k and n inputs, bounded insertion takes O(n * k) time and O(k) storage. When k is a fixed small constant, such as three, that is O(n) time and O(1) extra storage.
