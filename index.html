QUESTION 01

How does a mutable byte buffer work?

A []u8 is a mutable view over a sequence of bytes. The slice stores a pointer and a length: it knows where the storage begins and how many bytes are accessible, but not which bytes contain meaningful data.

var storage: [capacity]u8 = undefined;
const buffer: []u8 = &storage;

When building data incrementally, track the initialized portion with a separate cursor:

var used: usize = 0;

// Write new data starting at buffer[used].
// Advance used by the number of bytes written.

return buffer[0..used];

Core rule: buffer.len describes accessible storage. used describes initialized data.

QUESTION 02

How do I write bytes and strings into a buffer?

You can assign one byte by index:

buffer[i] = byte;

A byte slice cannot be replaced with a string using ordinary assignment. Copy the source bytes into a destination slice instead. @memcpy is appropriate when the source and destination do not overlap:

const text = "hello";
@memcpy(buffer[used .. used + text.len], text);
used += text.len;

Not valid: buffer[used .. used + text.len] = text;

Using text.len instead of a hard-coded length keeps the destination size tied to the actual source.

QUESTION 03

How do I expose only the initialized data?

Returning buffer returns its full length, including bytes you did not write. Slicing creates a view containing only the initialized prefix:

const initialized = buffer[0..used];
return initialized;

The returned slice still refers to the original storage. Slicing does not copy bytes or allocate memory, so that storage must remain valid while the returned slice is used.

QUESTION 04

Why does a buffer contain unexpected bytes?

Storage initialized with undefined has no usable initial value. Debug builds may fill untouched memory with a recognizable pattern such as 0xaa, but code must not rely on any particular value.

var storage: [capacity]u8 = undefined;

// Only storage[0..used] becomes valid after writing.

Unexpected trailing bytes usually mean that code exposed more of the buffer than it initialized. Read and return only the region known to contain valid data.

QUESTION 05

Why are buffer-formatting operations fallible?

A formatting function can run out of space, so it returns an error union such as error{NoSpaceLeft}![]u8. A function returning plain []const u8 must handle that error first.

const result = try std.mem.print(buffer, "{}", .{value});
return result;

Use try when the surrounding function can return the error. Use catch when it must recover, translate the error, or when a proven size invariant makes failure impossible.

Version note: Standard-library formatting names can differ between Zig versions. Check the documentation for the version used by the project.

QUESTION 06

How do I append multiple byte slices?

Keep a cursor and copy each selected piece at that cursor:

var used: usize = 0;

for (parts) |part| {
    @memcpy(buffer[used .. used + part.len], part);
    used += part.len;
}

return buffer[0..used];

A fixed-buffer writer is useful when many operations need to append or format data. It maintains the position and provides operations such as writeAll and print. An explicit cursor is often simpler for a small, fixed set of writes.

QUESTION 07

How do I reverse byte-string data?

To reverse bytes already written into the buffer, reverse only the initialized slice:

std.mem.reverse(u8, buffer[0..used]);
return buffer[0..used];

To copy from another slice in reverse order:

for (source, 0..) |_, i| {
    buffer[i] = source[source.len - 1 - i];
}

return buffer[0..source.len];

Text warning: This reverses bytes and is correct for ASCII. UTF-8 characters may contain multiple bytes and require Unicode-aware handling.

QUESTION 08

Which writing method should I choose?

Method Use it when
buffer[i] Writing or transforming one byte.
@memcpy Copying a complete non-overlapping byte slice.
copyForwards Preferring an explicit standard-library copy operation.
for loop Reversing, filtering, or changing bytes while copying.
Writer Performing many sequential or formatted writes.

Zig strings are byte slices. Copying a string copies its bytes and does not automatically add a null terminator.
